## 1. 项目背景

在电商促销场景中，秒杀活动因其"限时、限量、低价"的特点，能在短时间内聚集大量用户访问，对系统的并发处理能力提出极高要求。本项目旨在构建一套完整的高并发秒杀电商系统，覆盖从用户注册到秒杀下单的全链路功能，同时重点解决高并发场景下的核心技术难题。

---

## 2. 产品目标

| 目标          | 描述                                                    |
| ------------- | ------------------------------------------------------ |
| 功能完整性     | 覆盖用户、商品、库存、秒杀、订单五大核心模块                 |
| 高并发支撑     | 系统在 1000 并发下仍能稳定运行，不出现超卖                   |
| 数据一致性     | 库存扣减、订单创建保证最终一致性                             |
| 用户体验      | 秒杀下单响应时间 < 500ms（P95）                             |
| 可部署性      | Docker Compose 一键部署全栈环境                             |

## 4. 功能需求

### 4.1 用户模块

| 功能编号 | 功能名称   | 描述                                         | 优先级 |
| ------- | --------- | ------------------------------------------- | ----- |
| U-001   | 用户注册   | 用户通过用户名、密码、手机号完成注册              | P0    |
| U-002   | 用户登录   | 用户通过用户名和密码登录，返回 Token              | P0    |
| U-003   | 获取用户信息 | 登录用户获取个人基本信息                         | P1    |

#### 业务规则

- 用户名唯一，长度 4~20 字符
- 密码长度 6~32 字符，后端 BCrypt 加密存储
- 登录成功返回 JWT Token，有效期 24 小时
- 未登录用户无法参与秒杀

### 4.2 商品模块

| 功能编号 | 功能名称   | 描述                              | 优先级 |
| ------- | --------- | -------------------------------- | ----- |
| P-001   | 商品列表   | 分页展示所有商品信息                 | P0    |
| P-002   | 商品详情   | 展示单个商品详细信息，包含库存与秒杀状态 | P0    |

#### 业务规则

- 商品列表默认按创建时间倒序
- 商品详情需展示：名称、价格、秒杀价、库存数量、秒杀开始/结束时间
- 商品数据优先从 Redis 缓存读取

### 4.3 秒杀模块

| 功能编号 | 功能名称   | 描述                                   | 优先级 |
| ------- | --------- | ------------------------------------- | ----- |
| S-001   | 执行秒杀   | 用户点击秒杀按钮，系统处理抢购请求          | P0    |
| S-002   | 秒杀结果查询 | 用户查询自己的秒杀结果                    | P1    |

#### 业务规则

- 每个用户对同一商品只能秒杀一次（幂等控制）
- 秒杀必须在活动时间范围内
- 库存为 0 时直接返回售罄，不再访问数据库
- Redis 原子操作预减库存，扣减成功才进入后续流程
- 使用分布式锁防止并发超卖

#### 秒杀流程

```
用户请求 ---> 参数校验
              |
              v
         用户是否已秒杀过？ --Yes--> 返回"重复秒杀"
              |No
              v
         Redis库存预减 (DECR)
              |
              +---> 库存 < 0 ? ---> 恢复库存(INCR) + 返回"已售罄"
              |
              v
         获取分布式锁
              |
              +---> 失败 ? ---> 返回"系统繁忙"
              |
              v
         MySQL 扣减库存 (WHERE stock > 0)
              |
              +---> 影响行数 = 0 ? ---> 释放锁 + 返回"已售罄"
              |
              v
         创建秒杀订单
              |
              v
         释放分布式锁
              |
              v
         返回"秒杀成功" + 订单信息
```

### 4.4 订单模块

| 功能编号 | 功能名称   | 描述                          | 优先级 |
| ------- | --------- | ---------------------------- | ----- |
| O-001   | 创建订单   | 秒杀成功后自动创建订单            | P0    |
| O-002   | 订单列表   | 用户查看自己的所有订单            | P1    |
| O-003   | 订单详情   | 查看单个订单的详细信息            | P1    |

#### 业务规则

- 订单号全局唯一（雪花算法或 UUID）
- 订单状态：待支付 / 已支付 / 已取消
- 秒杀订单与普通订单关联

## 5. 非功能需求

### 5.1 性能需求

| 指标          | 目标值                       |
| ------------- | --------------------------- |
| 秒杀接口 QPS  | >= 1000                     |
| 秒杀响应时间   | P95 < 500ms                 |
| 数据库连接池   | 最大 50 连接                  |
| Redis 连接池   | 最大 100 连接                 |

### 5.2 可用性需求

- 系统可用性 >= 99.9%
- 库存数据最终一致性保证
- 秒杀失败不影响其他模块正常使用

### 5.3 安全需求

- 密码加密存储（BCrypt）
- JWT Token 鉴权
- 接口幂等性保证
- 防止刷单（同一用户限购）

## 6. 接口定义概览

| 方法   | 路径                    | 说明        | 是否需要登录 |
| ------ | ---------------------- | ---------- | ---------- |
| POST   | `/user/register`       | 用户注册    | 否          |
| POST   | `/user/login`          | 用户登录    | 否          |
| GET    | `/user/info`           | 用户信息    | 是          |
| GET    | `/product/list`        | 商品列表    | 否          |
| GET    | `/product/{id}`        | 商品详情    | 否          |
| POST   | `/seckill/{productId}` | 执行秒杀    | 是          |
| GET    | `/order/list`          | 订单列表    | 是          |
| GET    | `/order/{id}`          | 订单详情    | 是          |
